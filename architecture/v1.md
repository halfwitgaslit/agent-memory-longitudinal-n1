## 1. The contribution, locked

Three sub-contributions, one paper:

1. **Empirical** — first pre-registered, randomized, multi-framework N-of-1 deployment study of OSS agent-memory systems (Mem0, Letta, Hindsight, Cognee + null/random baselines) on a single developer's real coding-CLI work, with bootstrap-CI'd within-subject statistics on >100 sessions across Claude Code and Codex.
2. **Methodological — Per-Deployment Decay Calibration (PDDC).** Extension of FSRS (Anki, 2022+) replacing the binary recall signal with a 4-dimensional agent-task-success vector `(support_count, hit_rate, conflict_events, task_success_delta)`. Gradient-descent calibration over per-deployment session traces.
3. **Systems — Governed Cross-Worktree Memory Propagation (GCMP).** Extension of Letta Code MemFS (Feb 2026) with an explicit policy DSL for (a) selective parent→child inheritance at fork, (b) live cross-worktree querying without merge, and (c) per-fact calibrated promotion thresholds.

A fourth, secondary deliverable is **HONEST-Mem 15-field reproducibility reporting**, auto-generated from run metadata (a methods-section template enforced by code).

## 2. Why this conjunction is the right one (per Phase 0 novelty audit)

- **N1 (longitudinal n=1)** survives only with sharper framing: pre-registered + randomized multi-framework + coding-CLI + irreversibility-adapted N-of-1. Miessler (n=1 longitudinal personal AI infra, [un]prompted 2026) is the practitioner precedent we differentiate against; Kazdin 2011 / Kratochwill 2010 / Guyatt 1986 / CONSORT N-of-1 2015 are the methodological lineage we adapt.
- **N2 (cross-CLI bridging)** is dead-on-arrival (Mem0 ships official Claude Code + Codex integrations; AGENTS.md is a Linux Foundation standard). We *use* this substrate; we do not claim novelty on the bridging itself.
- **N3 (rule-lifecycle FSM with empirical decay)** survives only as *substrate + signal* novelty: FSRS owns calibrated decay; ISO 15489 owns lifecycle FSM; MACLA owns benchmark-calibrated multi-factor agent utility. PDDC's novelty is (a) calibration source = per-deployment trace, not benchmark grid search; (b) signal = 4D agent task success, not binary recall; (c) substrate = LLM agent rule memory.
- **N4 (worktree-aware memory)** survives only as a *policy/governance* layer atop Letta Code MemFS. GCMP's novelty is (a) selective parent→child injection at fork time with explicit scope rules (Letta ships full inheritance only), (b) live cross-worktree query without merge, (c) per-fact promotion calibrated against deployment outcomes.

## 4. Component specifications

### 4.1 Adapters

**Design contract:** lossless round-trip on metadata fields. Free-form text content (assistant messages, tool outputs) may be normalized but original is retained in `_raw` field.

**Claude Code JSONL schema (observed):**
- Record types: `user`, `assistant`, `attachment`, `queue-operation`, `last-prompt`, `ai-title`, `tool_use`, `tool_result`, `system`
- Common fields: `parentUuid`, `isSidechain`, `promptId`, `type`, `message`, `uuid`, `timestamp`, `userType`, `sessionId`
- Per-record `message.content` is a list of content blocks (text, tool_use, tool_result, thinking)

**Codex Rollout JSONL schema (observed):**
- Record types: `session_meta`, `turn_context`, `response_item`, `event_msg`
- `session_meta.payload`: id, cwd, originator, cli_version, source, model_provider, base_instructions
- `response_item.payload`: OpenAI Chat Completions message format
- `event_msg.payload`: status/lifecycle events

**Unified Turn schema (versioned v1):**
- `turn_id: str` — synthetic, sha1 of (session_id, ordinal, content_hash)
- `session_id: str`
- `ordinal: int` — position within session
- `role: Literal["user", "assistant", "system", "tool"]`
- `content: List[ContentBlock]`  — text, tool_use, tool_result, thinking
- `tool_events: List[ToolEvent]`
- `ts_utc: float`
- `model: str | None`
- `cli: Literal["claude_code", "codex"]`
- `worktree_id: str | None`  — derived from cwd parse
- `parent_branch: str | None`  — derived
- `_raw: Dict` — the original JSONL record(s) for round-trip integrity
- `_schema_version: str = "v1.0"`

Round-trip tests:
- Parse 5+ representative real sessions per CLI
- Re-serialize via per-CLI inverse adapter (or `_raw` preservation check)
- Assert deep equality on metadata fields

### 4.2 Memory backends

All backends implement `MemoryBackend(ABC)` with:
- `__init__(config: Dict, scope: Dict)`
- `add(turns: List[Turn], scope: Optional[Dict] = None) -> List[str]`  → memory IDs
- `search(query: str, k: int = 5, scope: Optional[Dict] = None) -> List[Memory]`
- `inspect() -> Dict`  → counts, latencies, errors, embedding-model id
- `decay_step(now_utc: float) -> Dict`  → counts updated/promoted/archived
- `lifecycle_promote(memory_id: str, new_state: str) -> bool`
- `clear() -> None`  → for between-cell reset

**Scope semantics:** scope is a free-form dict with reserved keys `user_id`, `project`, `worktree`, `branch`, `cli`. Backends are free to ignore keys they don't honor.

**Six backends in the bakeoff:**
1. `mem0_backend.py` — Mem0 v2 via `mem0.Memory.from_config` with fastembed + qdrant-on-disk + Anthropic Haiku for fact extraction.
2. `letta_backend.py` — letta-client to a local Letta server (port 8283); if unreachable, wrapper marks itself unhealthy and the arm is skipped with a logged reason.
3. `hindsight_backend.py` — hindsight-api with local SQLite + ONNX MiniLM embedder (no cloud calls).
4. `cognee_backend.py` — cognee with local config (LanceDB + Anthropic Haiku for graph extraction).
5. `null_backend.py` — no-op; returns `[]` for all searches. Negative control.
6. `random_backend.py` — returns k randomly chosen previously-added memories. Additional negative control.

**Smoke test (`tests/test_memory_smoke.py`):**
- Add a fixed synthetic corpus of 12 turns
- Query each backend with 3 fixed queries
- Assert every backend returns ≤ k results, returns valid `Memory` objects, persists across re-instantiation
- Record per-backend latency / spend / health

### 4.3 Novel layer — calibration/decay.py (PDDC)

Inputs:
- A trajectory of `(memory_id, timestamp, support_count, hit_rate, conflict_events, task_success_delta)` tuples per memory record from a deployment
- Initial FSRS-6 parameters (the published 21-parameter starting point)

Model:
- Per-memory state: `(stability, difficulty, last_review_ts, multi_dim_signal_history)`
- Retrievability: `R(t) = (1 + t / (9 · stability))^(-1)` (FSRS-6 form)
- Stability update on review: FSRS-6 update rule generalized — replace binary `recall ∈ {0, 1}` with continuous `task_success ∈ [0, 1]` and add three modulating signals
- Loss: weighted MSE between predicted retrievability and observed `hit_rate`, plus an auxiliary loss on `task_success_delta`

Optimizer: gradient descent on (21 + 4 extra) parameters via `scipy.optimize.minimize` (L-BFGS-B); per-deployment fit on held-in 70% of trajectory; held-out 30% for parameter selection.

Synthetic ground-truth test (`tests/test_calibration.py`):
- Generate synthetic memory trajectories from known true parameters
- Fit PDDC, assert recovered parameters within 10% of ground truth on >80% of dimensions
- Compare against FSRS-6 default-parameters baseline (assert PDDC strictly better on the synthetic data)

### 4.4 Novel layer — governance/cross_worktree.py (GCMP)

Three policies, each expressed as a small DSL on memory metadata:

```python
@dataclass
class InheritancePolicy:
    """At fork, which parent memories are injected into child worktree."""
    predicate: Callable[[Memory, ForkContext], bool]
    transform: Callable[[Memory], Memory]  # optional re-scoping
```

```python
@dataclass
class CrossQueryPolicy:
    """At runtime, when parent orchestrator searches across N live children."""
    enabled: bool
    sibling_visibility: Literal["all", "same_parent_branch", "none"]
    rerank_strategy: Literal["round_robin", "score_only", "score_with_sibling_boost"]
```

```python
@dataclass
class PromotionPolicy:
    """When does a child-worktree memory get promoted to the parent branch?"""
    min_support_count: int
    min_hit_rate: float
    max_conflict_events: int
    min_task_success_delta: float
    cooldown_seconds: int
```

Default policies (calibrated against the roomd corpus during Phase 2):
- Inheritance: predicate `lambda m, ctx: m.tag in {"convention", "constant", "decision"}`
- Cross-query: `score_with_sibling_boost`, sibling_visibility=`same_parent_branch`
- Promotion: `support_count ≥ 3, hit_rate ≥ 0.7, conflict_events ≤ 1, task_success_delta ≥ 0.05, cooldown ≥ 600s`

GCMP composes with any `MemoryBackend`. Test: `tests/test_governance.py` runs a 3-worktree synthetic fleet, verifies promotion threshold gates correctly, sibling visibility honored, fork inheritance is selective per policy.

### 4.5 Injection mechanism

**Claude Code skill (`injection/claude_code_skill/`):**
- `SKILL.md` with frontmatter `name: roomd-memory-retrieval`, description triggering on any coding session in roomd paths
- `retriever.py` invoked from the skill body: takes the session's first user prompt, calls the configured memory backend, returns top-k formatted as a `## Relevant prior knowledge` block
- Skill is installable via `~/.claude/plugins/`; install path scripted in Makefile

**Codex prompt hook (`injection/codex_prompt_hook/`):**
- Codex consults `AGENTS.md` in cwd. Hook writes a transient `AGENTS.md` augmentation at session start.
- For sessions already in flight, Codex doesn't re-read AGENTS.md, so the hook is best-effort. Document this asymmetry in the limitations section.
- Alternative: pre-pend memory to the user's first message via a small shell wrapper around `codex`.

**Verification test:** spin up a real Claude Code session, observe in the resulting JSONL that a `## Relevant prior knowledge` block appears in the system/user content. See `tests/test_injection_live.py` (manual run, not CI).

### 4.6 Eval harness

**Arm randomization (per E2 RBSD arxiv 2506.12654 switchback design):**
- Unit = a single "task" (a coherent user goal worked end-to-end). Tasks identified ad-hoc by Vector and logged via `phd/code/eval/task_log.py`.
- Arms: `{null, random, mem0, letta, hindsight, cognee}` (6 arms; Letta may be skipped if server unhealthy)
- Randomization: stratified by (time-of-day-bucket, project) with a per-bucket Latin-square switchback to control time-of-day variance (per E2 finding: 20% variance from time-of-day)
- Pre-registered seed: SHA256 of pre-registration commit hash → first 8 bytes → np.random.default_rng seed

**Metrics (`eval/metrics.py`):**
- `task_success_binary` — Vector's self-rating, 0 or 1, captured immediately after task
- `task_success_5pt` — 1-5 Likert
- `time_to_first_useful_output_s` — derived from JSONL
- `total_session_duration_s`
- `total_token_spend_usd` — derived from JSONL usage records
- `retry_count` — count of `tool_use_id` repeats with different params
- `had_to_remind_count` — count of user turns containing "as I told you" / "remember" / "you forgot" (regex; manual verification spot-check)
- `prior_knowledge_recall` — for sessions where memory was injected, did the assistant cite or use it? LLM-graded (3 judges, κ ≥ 0.7 vs Vector's spot-checks)
- `conflict_events` — count of memory entries flagged as contradicted by session outcome

**Stats (`eval/stats.py`):**
- Wilcoxon signed-rank for paired arm comparisons (since the same Vector experiences all arms)
- Bonferroni correction across all pairwise comparisons (15 pairs for 6 arms, so α=0.05/15)
- Bootstrap 95% CI on every reported mean (B=10K, BCa correction)
- Cohen's d AND Cliff's δ for every comparison
- MADCovar variance-reduction (arxiv 2506.20523) using time-of-day and project as covariates

**Pre-registration (`eval/pre_registration.py`):**
- Reads `eval/experimental_constants.py` (locked constants: arm list, metrics, stat tests, B, α)
- Computes SHA-256 hash; commits hash to git
- Pre-reg commit hash IS the priority timestamp

### 4.7 Reporting — honest_mem.py

Auto-generates 15-field methods section per HONEST-Mem template (from HMA-1's adversarial-review-driven methodology):

1. Substrate (corpus stats)
2. Population of interest (n=1, named subject with consent)
3. Pre-registration (commit hash + git URL)
4. Randomization scheme (seed source, design type)
5. Arms (named, with backend versions)
6. Metrics (operational definitions)
7. Statistical procedures (with package versions)
8. Multi-judge protocol (model IDs, κ scores)
9. Effect sizes (test type, reporting)
10. Decontamination certification (test set never used for tuning)
11. Spend ledger (cumulative, by-probe)
12. Ablation table pointer
13. Negative-controls outcomes
14. Datasheet pointer
15. Limitations (n=1 honest disclosure)

| `BOOTSTRAP_B` | 10000 | Standard for BCa CI |
| `ALPHA_RAW` | 0.05 | Standard |
| `ALPHA_CORRECTED` | 0.05 / 15 ≈ 0.00333 | Bonferroni for 15 pairwise comparisons |
| `JUDGES_N` | 3 | Multi-judge to derive κ; per-pair κ ≥ 0.7 required |
| `KAPPA_THRESHOLD` | 0.7 | Standard for "substantial agreement" |
| `PERMUTATION_N` | 6 | All orderings of a 3-element list for permutation-averaged pairwise |

| `EMBEDDING_MODEL` | BAAI/bge-small-en-v1.5 (initial); Qwen3-Embedding-8B-MLX (target) | fastembed defaults; MLX swap when Phase 2 starts |
| `VECTOR_STORE` | qdrant-on-disk (initial); LanceDB (eval may compare) | qdrant is mem0's default |

Any change to these constants requires a new pre-registration commit and a noted Phase reset.

## 6. Data flow (concrete timeline)

```
T=0    Vector starts a roomd task with memory arm assigned (skill reads env var ROOMD_MEM_ARM)
T=10s  Skill runs retriever.py → calls MemoryBackend.search(query=first_user_prompt, k=5)
T=11s  Top-5 memories formatted into a "## Relevant prior knowledge" system block
T=11s  Block injected; session proceeds
T=N    Session ends; JSONL flushed by Claude Code
T=N+1  Vector closes session and runs `phd-mem record-task --session=<id> --arm=<arm> --success=<0|1> --notes='...'`
T=N+2  eval/runner.py picks up new task records, computes metrics, appends to results JSONL
T=N+3  At any time: phd-mem stats → emits current paired-comparison table
```

## 7. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Letta server fails to install | Letta arm skipped; logged in `decisions/build_progress.md`; reported in paper limitations |
| Anthropic API rate-limit hit during eval | All LLM calls billed through `claude -p` (subscription); rate-limit handled by harness backoff |
| Vector forgets to record arm/outcome | Skill writes `~/.roomd/mem_arm_session_<id>.json` at injection time; outcome can be filled post-hoc |
| Time-of-day confound | Stratified switchback randomization |
| Order-of-arms confound | Pre-registered Latin square per stratum |
| Memory carryover between arms (irreversibility) | Each arm's MemoryBackend is `clear()`ed between cells when isolated mode is on; alternative additive mode is also pre-registered (both run, separately) |
| Judge bias | Permutation averaging + multi-judge with κ check |
| Scooped by Sandelin's planned 27-session follow-up | Pre-registration commit timestamp locks priority |
| Letta velocity (Feb 2026 → may ship selective inheritance during paper writing) | Document Letta version at pre-reg; treat as a baseline to compare against |
